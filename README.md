# AWS IAM / Organizations Portfolio

本リポジトリは、AWS Organizations / IAM Identity Center を中心に、
**組織レベルのアクセス制御と権限管理をどのように設計・検証すべきか**
をテーマとして作成したポートフォリオ群をまとめたものです。

個別の IAM 設計だけに依存せず、
Organizations レベルでのガードレール設計と、
Identity Center を用いた権限管理を組み合わせることで、
実運用に耐えるセキュリティベースラインを構築できるかを検証しています。

---

## なぜポートフォリオを分けているか

IAM / Organizations 周りの設計は、
一つのリポジトリにまとめると論点が混ざりやすく、
「何を検証したのか」が不明瞭になりがちです。

本ポートフォリオでは以下の考え方に基づき、
**役割ごとにリポジトリを分割**しています。

- Organizations / SCP：  
  そもそも「できないこと」を定義するガードレール
- IAM Identity Center：  
  ユーザーと権限を分離した入口設計
- 設計制約・限界：  
  実務で詰まりやすい境界条件の整理

---

## ポートフォリオ一覧

### 1. AWS Organizations / SCP 基本設計
**aws-org-scp-portfolio**  
AWS Organizations 配下における SCP の基本的な振る舞いと、
OU 単位での制御設計を検証したポートフォリオです。

SCP が「許可」ではなく
**実行可能な操作の上限を定義する仕組み**であることを、
実際の操作検証を通じて整理しています。

---

### 2. SCP Guardrails 設計・検証
**aws-org-scp-guardrails-portfolio**  
SCP を用いたガードレール設計に焦点を当て、
管理者権限であっても拒否される制御が
実際に機能するかを検証したポートフォリオです。

IAM 設計に依存しない
**組織レベルの予防統制**が成立することを確認しています。

---

### 3. IAM Identity Center / Permission Set 設計
**aws-iam-permission-set-portfolio**  
IAM Identity Center の Permission Set を用いて、
ユーザーと権限を分離したロールベース管理を設計・検証したポートフォリオです。

ユーザーと権限を疎結合に管理することで、
運用変更に強い権限設計が可能であることを整理しています。

---

### 4. IAM Identity Center の設計制限・限界検証
**aws-iam-identitycenter-design-limits**  
IAM Identity Center を実運用で利用する際に直面しやすい
設計制約や限界を整理・検証したポートフォリオです。

「何ができるか」だけでなく、
**どこまでを SCP や IAM 側で担うべきか**
という責任境界を明確にしています。

---

## 本ポートフォリオ全体で示していること

本ポートフォリオ群を通じて、

- IAM 設計だけではセキュリティは完結しないこと
- SCP による組織レベルのガードレール設計が前提条件になること
- Identity Center は入口管理として有効だが、万能ではないこと

を実際の操作検証を通じて確認しました。

権限を「どう与えるか」だけでなく、
**「そもそも実行させない設計」を先に置くことが、
安定したセキュリティと運用につながる**
という判断を示しています。
